# Agent Update — on-demand patrol contract

This file is for whichever AI is reading it, invoked by a human who pasted this repo
(or this file + `PATROL.md`) into a chat session. There is no automation that runs
this work — `.github/workflows/patrol.yml` only *generates the queue* below with plain
Node, on a schedule, for free, with no API key. **Turning that queue into actual data
edits is a job for you, right now, because a human asked you to do it.**

## Kickoff line (a human can paste this verbatim)

> Update Slow Variables from its patrol queue: read `PATROL.md` and `AGENT_UPDATE.md` in
> `<repo url>`, re-verify the stalest entries using your own web browsing/search,
> edit the JSON, validate, and open a PR.

If you were handed that line, keep reading — this document is the rest of your
instructions.

## 1. Read the queue

- **`PATROL.md`** — human-readable. Two sections that matter: **Kill-watch alerts**
  (theses whose `kill_watch.proximity` is at or past 0.8 — within roughly 20% of the
  stated trigger) and **Stale — needs re-verification** (every entity past its
  `review_after_months` window, ranked most-overdue first).
- **`patrol-queue.json`** — the same data, structured, if you'd rather parse it than
  the markdown. Fields: `kill_watch_alerts[]` and `stale_entities[]`, each with the
  entity id, registry, title, months overdue, the signal to check (with a URL if one
  exists), and current `cf`.

Both files are regenerated by `node scripts/patrol-report.js` (pure Node, no network,
no AI — it only computes overdue-ness from dates already in the data, it never tells
you what's actually true). If either file looks stale relative to the data, regenerate
it yourself before you start: `node scripts/patrol-report.js`.

## 2. Work the queue

**Kill-watch entries first.** For each one:
- Use your own web browsing/search to re-check the actual signal named in
  `kill_watch.signal` against its `trigger_value`.
- Update `kill_watch.current_value` and `kill_watch.as_of` (today's date) with what you
  actually found, sourced.
- Only update `kill_watch.proximity` if you have real information to justify a new
  0–1 estimate — it's an editorial judgment call, not a formula, but it must be
  grounded in something you just verified, not vibes.

**Then the stale entities**, in the order `PATROL.md` lists them (most-overdue first —
but use judgment; if a human asked for "the top 5" or gave you a time budget, respect
that instead of trying to clear the whole queue in one pass):
- Use your own web browsing/search on the "signal to check" column/field. If it links
  to a URL, start there.
- If the entity has a non-empty `signals[]` array, append a new entry to the relevant
  signal's `log[]`: `{"date": "<today>", "note": "<one-line finding>"}`, and update that
  signal's `last_checked` to today.
- Update the entity's content field(s) and `as_of` (to today) **only** when you found a
  real, dated, sourced confirmation or correction — never touch the content field on
  a guess.

**The rules governing every edit are in [`UPDATE_PROTOCOL.md`](UPDATE_PROTOCOL.md) —
read it now if you haven't. In short:**
- Never invent a number. If you can't verify something, say so and downgrade `cf` to
  `"md"` (or `"lo"` if you're really unsure) instead of leaving it at `"hi"` on a guess,
  or leave the field untouched if you have nothing better than what's already there.
- IDs are append-only — never rename, remove, or renumber an existing entity.
- Data edits only ever go in `/data/*.json`. Don't touch `/js` or `/css` for this task.
- "Still true as of today" is a completely valid outcome for an entity — you don't have
  to change content to have done useful work; a fresh signal log entry is real progress.

## 3. Close out

1. Run `node js/validate.js` — must pass with 0 errors. Fix your edit if it doesn't;
   don't work around the validator.
2. Regenerate the queue: `node scripts/patrol-report.js`. Commit the refreshed
   `PATROL.md` and `patrol-queue.json` alongside your data edits so the queue reflects
   what you just did (entities you fixed should drop off the stale list; entities you
   couldn't verify stay on it, honestly).
3. Bump `data/meta.json`: increment `version`, and add a `changelog` entry dated today
   noting **which AI did the pass** (e.g. `"by": "claude-sonnet-4-6"` or
   `"by": "gpt-5-thinking"` — whatever you are) and a one-line summary of what changed.
4. **Commit on a branch and open a PR.** Only commit straight to `main` if the human
   operating you explicitly tells you to in this conversation — by default, treat this
   exactly like any other data change: a branch, a commit, a PR for a human to review.
   There is no bot identity for this path; use your own judgment about commit authorship
   the same way you would for any other task a human asked you to do directly.

## 4. If a human also asks you to refresh HELM

HELM (the private personal lens, see `pilot.example.json` for its schema) is a
separate, optional pass — only do this if the human explicitly asks, and only if they
have a local `pilot.json` (never ask them to paste its contents into a shared/public
context; treat it as sensitive). It is not part of the default patrol/kickoff flow
above.

If asked:
1. Read the human's `pilot.json` (they'll need to share it with you directly, e.g. by
   opening it in this session — it's gitignored, so it won't be in the repo checkout).
2. Re-run the same kind of derivation `js/engine.js`'s `buildHelm*` functions do —
   `helmWindowMatches()` (skill-to-window text matches against open/forming windows),
   the leverage-point ranking, the asymmetric-bet cards, and the inversion check against
   closing/closed windows — but with real judgment instead of the app's crude substring
   matching. You have the context to catch matches the simple keyword overlap misses,
   and to reject matches it wrongly surfaces.
3. Every HELM recommendation must still cite the exact node/window/constraint id it
   comes from and state a falsifier — that rule doesn't relax just because a human is
   asking directly instead of the in-app heuristic doing it. If you can't trace a
   recommendation to a specific entity in `/data/*.json`, don't make it.
4. HELM output is for the human, not for a commit — there's nothing to check in here.
   `pilot.json` stays local and gitignored; don't suggest committing it or any file
   derived from it.

## What this is not

This is not a background job. Nothing in this repo calls out to any AI API, ever — grep
`.github/workflows/` if you want to confirm that yourself. The weekly `patrol.yml`
workflow is 100% deterministic Node with zero dependencies; it produces the queue you
just read and nothing else. The actual re-verification — the part that requires
judgment, web access, and the ability to decide "is this still true" — only happens
when a human like the one who gave you the kickoff line above decides it's worth an
AI session's time.
